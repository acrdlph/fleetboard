⌁ running a fleet — the runbook

fleetboard is an agent harness — a read-only observer with an opt-in control plane: watching never touches your sessions; acting (chat, resume, dispatch, finish) only happens on an explicit click, and always by talking to a terminal. This page is the operating manual — what the statuses mean, how a mission flows, and how to close one out cleanly.

§1 reading the board

A worktree card is ◇ FREE when it has no live process and no session written to recently — that's the whole definition. There is no button that frees a worktree; it happens by itself when the conditions are met. BUSY means something is actively working; NEEDS YOU means at least one session wants a human.

● WORKINGthe transcript — or a background subagent's — was written seconds ago. The agent may look idle at its prompt while subagents still run; the board is not wrong, look for the ⚙ subagents running tag.
▲ NEEDS ANSWERa live process has a question waiting for you. Open ✉ chat and answer it.
⛔ LIMIT HITparked on an exhausted account — the binding limit and reset countdown are shown. ▶ resume arms itself when the limit resets.
■ BLOCKEDa live process is stuck on an unresolved tool call (usually a permission prompt in its terminal).
◆ YOUR TURNthe turn is finished and the agent sits idle at its prompt. Read the result, then continue it or close it out.
○ ENDEDno live process, transcript quiet. Hidden by default — toggle “show ended sessions”.

§2 launching a mission

🚀 new mission dispatches a tmux-hosted agent. Your text reaches the agent verbatim; the optional router only picks where it runs — worktree and account default to the best free / most-headroom picks, respecting each account's reserve buffer (set on the limits page). Dispatched sessions are named mission-<worktree>-<time> on the fleet tmux socket: tmux -L fleet ls shows them, tmux -L fleet attach -t <name> gives you the real terminal.

§3 while it runs

The board refreshes every 5 s and sorts by attention. ✉ chat shows the transcript and types straight into the agent's terminal — it is exactly as if you typed there yourself. ⌖ pid jumps to the terminal window. On the map, a pulsing node is working; click any node for actions (⌖ focus · ✓ finish).

gotcha: the map's click-toast says “no live process — free for a new agent” whenever the directory has no process. A session can still be ● WORKING through background subagents writing transcripts. When the two disagree, trust the status, not the toast.

§4 closing out a mission

One button: ✓ finish — on every worktree card, and in a map node's panel (click the node). It arms on the first click and runs on the second, so a stray click does nothing. Like every fleetboard action, it works by talking to a terminal:

  • agent alive → it receives the closeout brief: land the branch on the trunk, tidy the worktree, report back. Once it reports landed, ✓ finish again closes its terminal.
  • terminal already gone → a small closeout agent is dispatched into the worktree to do the same.
  • everything landed, agent idle → /exit is typed and the terminal closes.

Then wait a few seconds — the session flips to ○ ENDED and the card returns to ◇ FREE by itself; free is a state the board observes, not one you set.

what the closeout brief asks the agent to do — the manual runbook
  1. Settle background work. Wait for (or stop) any background agents and workflows — they keep writing transcripts after the prompt goes idle, which keeps the card honestly ● WORKING.
  2. Commit what matters. Commit remaining meaningful work; drop scratch files.
  3. Land the branch. Merge the trunk into the branch, resolve any conflicts, push the branch, then push the result to the trunk. Pushing a feature branch only backs it up — work reaches the trunk through a merge. Check: git merge-base --is-ancestor HEAD origin/main && echo landed.
  4. Reset the worktree. git switch main && git pull, so the next mission starts from a fresh trunk.
  5. Report. One line on what landed. Doing it all by hand instead: after these steps, end the session with /exit or tmux -L fleet kill-session -t mission-<worktree>-<time>.

§5 limits & accounts

The limits page shows every account's headroom; bars show used capacity. A reserve buffer keeps that percentage of the weekly limit free from auto-dispatch — hand-picked launches can still use it. A session parked on ⛔ LIMIT HIT shows the binding limit and its reset time; the ▶ resume button on its row arms itself when the reset passes and types continue into its terminal.
