# ADR 0009 — The versioned API starts at `/api/v1`

**Date:** 2026-07-21 · **Status:** Accepted

## Context

`ARCHITECTURE.md` specified `/api/v2` while `API.md` specified `/api/v1`. The disagreement
survived a coherence pass, and it is baked into a 3,454-line contract, so it had to be settled
before any code was written against it.

The argument for `v2`: the existing unversioned surface (`/api/state`, `/api/dispatch`, …) is
*de facto* v1, so naming the new one v1 invites confusion about which is which.

## Decision

**`/api/v1`.**

The existing surface was never *named* v1. It is unversioned legacy, and `API.md` documents it
as exactly that in its Legacy section, with a per-path mapping. Starting at v2 avoids a
collision that does not exist, and does so at a real cost: it implies there was a v1 a client
could have targeted, which is misleading to anyone arriving fresh.

It is also the cheaper edit — v1 was already used throughout the contract document, so 27
references in `ARCHITECTURE.md` and `ROADMAP.md` moved rather than 3,454 lines of `API.md`.

## Consequences

- The legacy unversioned paths keep working and are documented as legacy, not as v1. Their
  compatibility guarantee is stated in `API.md`.
- The Swift client, the SSE stream and the push pipeline all address `/api/v1/…`.
- If a genuinely breaking revision is ever needed, it is `/api/v2`, and the numbering means what
  everyone expects.

## Alternatives rejected

- **`/api/v2`** — see Context. Rejected as cargo-culting around a non-existent collision.
- **Stay unversioned, negotiate by header.** More flexible in principle, but harder to debug by
  hand (`curl` against a path is the project's normal idiom) and harder to cache. Rejected.
